# JudgeMesh 四节点服务器集群部署指南

本文给出一套和当前仓库 `infra/k8s/`、`scripts/k8s-bootstrap.sh`、`infra/helm/values/` 对齐的 4 台服务器部署方案，目标是把 JudgeMesh 部署成“业务多副本 + dispatcher 选主 + worker 多实例”的演示集群。

本文按 2026-05-20 仓库状态编写。当前业务层高可用重点在：

- `gateway` 多副本
- `submit-service` 多副本
- `judge-dispatcher` 3 副本 + etcd 选主
- `judge-worker` 多实例 + 故障摘除

当前中间件仍以单实例为主，属于课程演示可接受范围，答辩时应明确说明。

## 1. 节点规划

建议 4 台 Ubuntu 22.04 服务器，最低 `2C4G`，更稳妥是 `4C8G`。

| 节点 | 角色 | 主要承载 |
| --- | --- | --- |
| `node-1` | control-plane | kube-apiserver / scheduler / controller-manager |
| `node-2` | worker | gateway / frontend / user-service / problem-service |
| `node-3` | worker + judge | submit-service / judge-dispatcher / judge-worker |
| `node-4` | worker + judge | judge-worker / 可观测 / 中间件 |

Judge 节点需要单独打标签和 taint，便于 `judge-worker` 定向调度。

## 2. 系统准备

4 台机器都执行：

```bash
sudo swapoff -a
sudo sed -i.bak '/ swap / s/^/#/' /etc/fstab

cat <<'EOF' | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

sudo modprobe overlay
sudo modprobe br_netfilter

cat <<'EOF' | sudo tee /etc/sysctl.d/99-kubernetes-cri.conf
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward = 1
EOF

sudo sysctl --system
```

安装运行时和 K8s 基础工具：

```bash
sudo apt-get update
sudo apt-get install -y containerd apt-transport-https ca-certificates curl gnupg lsb-release

sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml >/dev/null
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml
sudo systemctl restart containerd
sudo systemctl enable containerd
```

安装 `kubeadm`、`kubelet`、`kubectl`：

```bash
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.31/deb/Release.key | \
  sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.31/deb/ /' | \
  sudo tee /etc/apt/sources.list.d/kubernetes.list

sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl
sudo apt-mark hold kubelet kubeadm kubectl
```

所有节点确认：

```bash
containerd --version
kubeadm version
kubectl version --client
```

## 3. 初始化 control-plane

在 `node-1` 克隆仓库后执行 dry-run：

```bash
cd /path/to/JudgeMesh
scripts/k8s-bootstrap.sh
```

确认参数无误后正式初始化：

```bash
cd /path/to/JudgeMesh
CNI_PROVIDER=calico scripts/k8s-bootstrap.sh --confirm
```

脚本会：

- 执行 `kubeadm init`
- 配置当前用户 `~/.kube/config`
- 安装 Calico
- 输出 join 命令到 `/tmp/judgemesh-kubeadm-join.sh`

检查：

```bash
kubectl get nodes -o wide
kubectl get pods -A
```

## 4. 加入另外 3 台机器

在 `node-2`、`node-3`、`node-4` 执行 `node-1` 输出的 join 命令，例如：

```bash
sudo kubeadm join <control-plane-ip>:6443 \
  --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash>
```

回到 `node-1` 检查：

```bash
kubectl get nodes
```

给 judge 节点打标签和 taint：

```bash
kubectl label node node-3 role=judge
kubectl label node node-4 role=judge

kubectl taint node node-3 role=judge:NoSchedule
kubectl taint node node-4 role=judge:NoSchedule
```

验证：

```bash
kubectl get nodes --show-labels
```

## 5. 安装 Helm、Ingress、证书与扩缩容组件

在 `node-1`：

```bash
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
```

安装 `ingress-nginx`：

```bash
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

helm upgrade --install ingress-nginx ingress-nginx/ingress-nginx \
  -n ingress-nginx --create-namespace
```

安装 `cert-manager`：

```bash
helm repo add jetstack https://charts.jetstack.io
helm repo update

helm upgrade --install cert-manager jetstack/cert-manager \
  -n cert-manager --create-namespace \
  --set crds.enabled=true
```

安装 `keda`：

```bash
helm repo add kedacore https://kedacore.github.io/charts
helm repo update

helm upgrade --install keda kedacore/keda \
  -n keda --create-namespace
```

## 7. 准备业务密钥

当前仓库使用 `judgemesh-secrets` 作为应用主要 secret。

至少创建：

```bash
kubectl create namespace judgemesh || true
kubectl create namespace judgemesh-infra || true
kubectl create namespace judgemesh-observe || true

kubectl create secret generic judgemesh-secrets \
  --from-literal=MYSQL_USER=judgemesh \
  --from-literal=MYSQL_PASSWORD='<replace-me>' \
  --from-literal=RABBITMQ_USER=judgemesh \
  --from-literal=RABBITMQ_PASSWORD='<replace-me>' \
  --from-literal=JWT_SECRET='<replace-with-32-bytes-or-more>' \
  -n judgemesh
```

如果你要调整 `app-secrets` 与 `infra-secrets` 模板，先编辑：

- `infra/k8s/base/app-secrets/secret-template.yaml`
- `infra/k8s/base/infra-secrets/secret-template.yaml`

再应用 overlay。

## 8. 部署 JudgeMesh

开发联调用 `dev`：

```bash
kubectl apply -k infra/k8s/overlays/dev
```

4 节点演示推荐直接用 `prod`：

```bash
kubectl apply -k infra/k8s/overlays/prod
```

建议部署后按顺序检查：

```bash
kubectl get pods -A
kubectl get svc -A
kubectl get ingress -A
kubectl get scaledobject -A
```

重点确认：

- `judge-dispatcher` 有 3 个 Pod
- `judge-worker` 被调度到 `role=judge` 节点
- `gateway`、`submit-service` 都至少 2 副本
- `judgemesh-observe` 名字空间里的 Grafana/Prometheus 正常运行

## 9. 集群上线后验证

### 9.1 基础接口验证

先转发关键端口：

```bash
scripts/port-forward.sh
```

然后检查：

```bash
curl http://127.0.0.1:8080/actuator/health
curl http://127.0.0.1:8084/api/admin/dispatcher/status
BASE_URL=http://127.0.0.1:8080 scripts/smoke-test.sh
```

### 9.2 生成演示数据

导入 demo 题：

```bash
python3 scripts/import-problems.py --service http://127.0.0.1:8080 --limit 30
```

生成用户、普通提交、比赛提交：

```bash
node scripts/seed-demo-data.mjs
```

如果要生成慢任务，便于演示 worker 切换和排队：

```bash
INCLUDE_SLOW=true TOTAL=24 CONTEST_USERS=8 node scripts/seed-demo-data.mjs
```

如果要做更明显的分布式调度演示：

```bash
TOTAL=60 CONCURRENCY=30 node scripts/demo-distributed-load.mjs
```

### 9.3 高可用演示检查

查看当前 leader 与 worker 池：

```bash
curl http://127.0.0.1:8084/api/admin/dispatcher/status | jq
```

在前端首页可以直接看到：

- 当前 leader
- dispatcher 模式
- worker 健康状态
- 全局榜单实时推送

## 10. 故障演示建议

### 杀 dispatcher leader

```bash
kubectl get pods -n judgemesh -l app=judge-dispatcher
kubectl delete pod <leader-pod> -n judgemesh
```

期望结果：

- 其他 dispatcher 副本抢到 leader
- 新提交仍能继续分发

### 杀一个 worker

```bash
kubectl get pods -n judgemesh -l app=judge-worker -o wide
kubectl delete pod <worker-pod> -n judgemesh
```

期望结果：

- dispatcher 状态页中的该 worker 会短暂不可用或被拉黑
- 后续提交会切到其他健康 worker
- Deployment 自动拉起新的 worker Pod

### Chaos Mesh

如果已安装 Chaos Mesh，可直接使用：

```bash
kubectl apply -f infra/chaos/kill-leader.yaml
kubectl apply -f infra/chaos/kill-worker.yaml
kubectl apply -f infra/chaos/mq-down.yaml
```

## 11. 当前限制

- `MySQL`、`Redis`、`RabbitMQ`、`Nacos`、`MinIO` 仍偏单实例，更像“业务层高可用 + 中间件简化版”。
- `judge-worker` 依赖 `privileged: true` 运行 isolate，这是当前课程实现的已知约束。
- 若机器资源不足，优先下调 `judge-worker`、`gateway`、`submit-service` 副本数，再调整可观测组件资源。

## 12. 推荐答辩演示顺序

1. 打开前端首页，展示当前 leader 与 worker 池。
2. 提交一批代码，观察实时状态和 `judgedByWorker` 分布。
3. 打开比赛页，展示榜单实时变化。
4. 删除一个 worker Pod，继续提交，展示自动切换。
5. 删除 leader Pod，展示 dispatcher 重新选主。
6. 打开 Grafana 或 SkyWalking 展示指标和链路。
