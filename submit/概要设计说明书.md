## 1 引言

### 1.1 编写目的

本文档用于说明 JudgeMesh 分布式在线判题系统的概要设计方案，明确系统总体架构、运行环境、处理流程、模块结构、接口安排、运行控制、核心数据结构和出错处理策略。本文档承接《软件需求说明书》，为后续详细设计、编码实现、测试计划、部署实施和课程答辩提供设计依据。

### 1.2 范围

待开发系统名称为 JudgeMesh 分布式在线判题系统。任务提出者为武汉大学《分布式软件原理与技术》课程项目组，开发者为 JudgeMesh 五人项目组，最终用户包括学生、出题人、管理员、运维人员和课程答辩评审。系统运行单位为课程项目演示环境，部署目标为 Kubernetes 集群。

本文档覆盖以下概要设计范围：

- Web 前端、Gateway、业务微服务、判题调度器、判题 Worker、基础设施和可观测性组件。
- 提交到判题的异步流水线。
- 比赛、排行榜、WebSocket 推送和封榜机制。
- Nacos、Sentinel、Seata、RabbitMQ、Redis、MySQL、MinIO、etcd、Kubernetes 等组件的协作。
- 故障场景下的降级、恢复和维护设计。

## 2 总体设计

### 2.1 需求规定

系统需满足《软件需求说明书》中定义的 P0 和主要 P1 需求。概要设计中重点覆盖以下输入、处理和输出。

#### 2.1.1 主要输入

| 输入类别 | 典型输入 | 来源 |
| --- | --- | --- |
| 用户认证输入 | 用户名、邮箱、密码、JWT、Refresh Token | 浏览器 |
| 题目输入 | 题面、标签、时限、内存、测试用例文件 | 出题人 |
| 提交输入 | problemId、contestId、language、code | 学生 |
| 比赛输入 | 比赛时间、题目集合、封榜时间、报名用户 | 管理员或出题人 |
| 运维输入 | 配置、限流规则、混沌实验、发布命令 | 管理员或运维 |

#### 2.1.2 主要处理

- 认证与授权：Gateway 验签 JWT，下游服务根据角色控制访问。
- 题目管理：problem-service 管理题目元数据，MinIO 存储用例文件。
- 提交入队：submit-service 写提交记录并投递 RabbitMQ。
- 调度派发：judge-dispatcher 通过 etcd 选主，仅 leader 消费任务并选择 Worker。
- 沙箱判题：judge-worker 使用 isolate 编译运行代码，生成 JudgeResult。
- 结果回写：submit-service 更新 MySQL、Redis 排行榜并通过 WebSocket 推送。
- 观测与恢复：Prometheus、Grafana、SkyWalking、Loki 和 AlertManager 观察系统状态。

#### 2.1.3 主要输出

| 输出类别 | 典型输出 |
| --- | --- |
| 用户输出 | token、用户资料、权限错误 |
| 题目输出 | 题目列表、题目详情、测试用例 manifest |
| 提交输出 | submissionId、判题状态、耗时、内存、错误信息 |
| 比赛输出 | 比赛列表、报名结果、实时排行榜、封榜状态 |
| 运维输出 | 指标大盘、链路追踪、日志、告警、混沌实验结果 |

#### 2.1.4 主要性能要求

| 指标 | 目标 |
| --- | --- |
| 提交接口同步阶段 P99 | 小于 200 ms |
| 判题端到端 P99 | 小于 8 s |
| 排行榜查询 P99 | 小于 50 ms |
| 常态提交吞吐 | 大于 100 QPS |
| 比赛提交峰值 | 大于 500 QPS，同步入队不崩溃 |
| dispatcher 选主恢复 | 10 s 内 |
| Worker 宕机任务重派 | 30 s 内 |

| 节点 | 角色 | 主要负载 |

| node-worker-1 | 通用业务节点 | Gateway、业务服务、可观测性 |
| node-worker-2 | 判题节点 | judge-worker，带 `role=judge` 标签 |
| node-worker-3 | 混合节点 | judge-worker、中间件、可观测性 |

### 2.3 基本设计概念和处理流程

#### 2.3.1 设计原则

1. 单一职责：每个微服务管理自己的核心数据，不跨服务直接写库。
2. 松耦合：同步读取使用 OpenFeign，异步解耦使用 RabbitMQ。
3. 基础设施抽象：服务通过 Spring Cloud Alibaba、Kubernetes Service 和配置中心使用基础设施。
4. 可观测优先：核心服务从初期暴露指标、日志和链路追踪。
5. 可降级：Worker、MQ、下游服务异常时优先保护系统入口和数据完整性。
6. 范围控制：本期不做讨论区、题解、SPJ、交互题等非核心能力。

#### 2.3.3 提交判题处理流程

```text
1. 学生在前端选择题目、语言并提交代码。
2. Gateway 验证 JWT、执行限流并路由到 submit-service。
3. submit-service 校验题目、比赛、用户状态和防重提交。
4. submit-service 写入 submission，状态为 PENDING。
5. submit-service 投递 JudgeTask 到 RabbitMQ。
6. judge-dispatcher leader 消费任务。
7. dispatcher 从 Kubernetes 或本地缓存获取 Ready Worker。
8. dispatcher 按负载和健康状态选择 Worker，写 etcd inflight。
9. Worker 下载测试用例，初始化 isolate。
10. Worker 编译和运行用户代码，解析 isolate meta。
11. Worker 生成 JudgeResult 并回写 submit-service。
12. submit-service 更新 submission 状态。
13. submit-service 更新 Redis 排行榜和 Pub/Sub。
14. WebSocket 向浏览器推送判题状态和排行榜变化。
```

#### 2.3.4 一致性策略

系统不采用单一一致性策略，而是按数据域区分。

| 数据域 | 一致性级别 | PACELC 取舍 | 设计说明 |
| --- | --- | --- | --- |
| 用户账号和密码 | 强一致 | PC/EC | 安全相关，写 MySQL |
| 用户积分扣减 | 强一致 | PC/EC | Seata AT 演示跨服务事务 |
| 题目元数据 | 强一致 + 缓存 | PC/EL | 写少读多，写后失效缓存 |
| 测试用例文件 | 强一致 | PC/EC | MinIO 私有存储 |
| 提交记录 | 强一致 | PC/EC | 成绩数据必须可靠 |
| 判题结果回写 | 最终一致 | PA/EL | MQ + 重试，允许短暂 Pending |
| 排行榜 | 最终一致 | PA/EL | Redis ZSet，允许秒级滞后 |
| WebSocket 推送 | 至多一次 | PA/EL | 推送失败由客户端拉取兜底 |

### 2.6 人工处理过程

| 人工处理过程 | 说明 |
| --- | --- |
| 题目审核 | 出题人创建题目后，管理员可检查题面、样例和测试用例再发布 |

| 比赛解榜 | 封榜结束后管理员手动解榜并发布最终排名 |

### 2.7 尚未解决的问题

| 问题 | 当前处理 | 后续方向 |
| --- | --- | --- |
| isolate 容器需要 privileged 权限 | 课程演示集群接受该风险 | 生产环境改用更细粒度 capabilities 或 gVisor |
| 中间件单实例可用性不足 | 通过备份、告警和文档说明缓解 | 后续升级 MySQL 主从、Redis Cluster、RabbitMQ 集群 |
| Go Worker 与 Java 链路追踪整合有限 | Worker 暴露 traceId 和 Prometheus 指标 | 后续接入 OpenTelemetry |
| SPJ 和交互题不支持 | 本期明确不做 | 后续通过 Runner 和 Comparator 插件扩展 |
| 大规模比赛容量未完全验证 | 本期压测到课程目标容量 | 后续用更大集群和更真实负载验证 |

#### 3.1.2 交互要求

- 登录后由前端在请求头携带 `Authorization: Bearer <token>`。
- 提交代码后立即展示 `PENDING` 状态，并订阅 `/ws/submission/{id}`。
- WebSocket 断线时前端自动重连，并通过 REST 拉取最新状态兜底。
- 表单错误需要显示服务端返回的错误码和友好文案。
- 排行榜首次进入拉取全量，之后接收 delta 更新。

## 4 运行设计

### 4.1 运行模块组合

#### 4.1.1 用户登录运行组合

```text
frontend -> gateway -> user-service -> users_db / Redis session
```

运行说明：

1. 前端提交登录表单。
2. Gateway 路由到 user-service。
3. user-service 校验密码，签发 token。
4. refresh token 写入 Redis。

#### 4.1.2 题目浏览运行组合

```text
frontend -> gateway -> problem-service -> Redis cache -> problems_db
```

运行说明：

1. 前端请求题目列表或详情。
2. problem-service 先读 Redis。
3. 缓存未命中时查询 MySQL，回填 Redis。

#### 4.1.3 提交判题运行组合

```text
frontend -> gateway -> submit-service -> user-service / problem-service
  -> submits_db -> RabbitMQ -> judge-dispatcher -> etcd -> judge-worker
  -> MinIO -> isolate -> submit-service -> Redis -> WebSocket -> frontend
```

运行说明：

1. 同步阶段以低延迟完成校验、入库和入队。
2. 异步阶段由 dispatcher 和 Worker 完成判题。
3. 结果回写后通过 WebSocket 推送。

#### 4.1.4 比赛排行榜运行组合

```text
submit-service -> Redis ZSet -> Redis Pub/Sub -> WebSocket -> frontend
```

运行说明：

1. AC 后 submit-service 更新 Redis ZSet。
2. 若未封榜，发布排行榜 delta。
3. WebSocket 服务向订阅者推送。

#### 4.1.5 混沌演示运行组合

```text
admin frontend -> gateway -> admin endpoint / ChaosMesh
  -> Kubernetes PodKill / NetworkDelay
  -> Prometheus / Grafana / AlertManager
```

运行说明：

1. 管理员触发实验。
2. ChaosMesh 注入故障。
3. 系统通过重试、选主、限流或 HPA 恢复。
4. Grafana 展示指标变化。

### 4.2 运行控制

#### 4.2.1 启动控制

首次部署顺序：

```text
1. Kubernetes 集群就绪。
2. 安装 ingress-nginx 和 cert-manager。
3. 安装 Prometheus、Grafana、SkyWalking、Loki。
4. 安装 ChaosMesh。
5. 部署 MySQL、Redis、MinIO、etcd、RabbitMQ、Nacos、Seata。
6. 导入 Nacos 业务配置。
7. 部署 gateway、user、problem、submit、dispatcher、worker。
8. 部署 Ingress。
9. 执行 smoke test 验证全链路。
```

#### 4.2.2 发布控制

- 代码合并到主分支后由 CI 运行测试、构建镜像和推送镜像仓库。
- Kustomize 或 Helm 更新镜像 tag。
- Kubernetes Deployment 使用 RollingUpdate。
- readiness probe 通过后新副本接流量。
- 失败时使用 `kubectl rollout undo deployment/<name>` 回滚。

#### 4.2.3 配置控制

| 配置类型 | 存储位置 | 更新方式 |
| --- | --- | --- |
| 默认配置 | 应用镜像内 `application.yml` | 随版本发布 |
| 业务可调配置 | Nacos | 热更新 |
| K8s 地址和资源配置 | ConfigMap / manifest | `kubectl apply` |
| 密钥 | K8s Secret | 手动或密钥管理工具注入 |
| 限流熔断规则 | Sentinel / Nacos | 控制台或配置更新 |

#### 4.2.4 故障控制

| 故障 | 控制方式 |
| --- | --- |
| Worker 宕机 | dispatcher 超时后重新派发，K8s 拉起新 Pod |
| dispatcher leader 宕机 | etcd Lease 过期，follower 竞选新 leader |
| 业务服务实例异常 | readiness 失败后流量摘除 |
| MQ 堆积 | HPA 扩容 Worker，必要时 Gateway 限流 |
| 下游服务慢调用 | Sentinel 熔断并执行 fallback |

### 4.3 运行时间

| 运行过程 | 目标耗时或资源占用 |
| --- | --- |
| 登录接口 | P99 小于 200 ms |
| 题目详情读取 | P99 小于 100 ms，缓存命中更低 |
| 提交同步阶段 | P99 小于 200 ms |
| 判题端到端 | P50 小于 2 s，P99 小于 8 s |

| Prometheus 抓取 | 应用指标 15 s，主机指标 30 s |

### 6.2 补救措施

#### 6.2.1 后备技术

| 数据或服务 | 后备措施 |
| --- | --- |
| MySQL | 每日 mysqldump 到 MinIO，必要时恢复 |
| Redis 排行榜 | 从 MySQL submission 重建 |
| RabbitMQ | durable queue 和 persistent message，DLQ 保存最终失败任务 |
| MinIO 测试用例 | MinIO 版本控制和定期备份 |
| Nacos 配置 | Nacos 版本回滚，manifest 中保留默认配置 |
| 应用镜像 | 镜像仓库保留 git-sha tag，可回滚到旧版本 |

#### 6.2.2 降效技术

| 故障场景 | 降效策略 |
| --- | --- |
| Worker 池不足 | 提交继续入队，网关降低提交速率 |
| Redis 不可用 | 排行榜暂不可用，提交结果仍写 MySQL |
| WebSocket 不可用 | 前端改用轮询查询提交状态 |
| user-service 慢调用 | Sentinel fallback 返回占位公开资料 |
| problem-service 缓存失效 | 直接查 MySQL，必要时限流 |
| SkyWalking 或 Loki 不可用 | 不影响业务，保留本地标准输出日志 |

#### 6.2.3 恢复及再启动技术

| 故障 | 恢复方式 |
| --- | --- |
| 业务服务崩溃 | Kubernetes 自动重启，readiness 通过后接流量 |
| Worker 崩溃 | dispatcher 超时重派，Deployment 拉起新 Pod |
| dispatcher leader 崩溃 | etcd Lease 过期后 follower 抢锁 |
| 发布失败 | `kubectl rollout undo` 回滚 |
| 配置错误 | Nacos 配置版本回滚 |
| 数据库迁移失败 | Flyway repair 或回滚脚本，停止相关服务写入 |
| 队列堆积 | HPA 扩容 Worker，必要时限制提交入口 |

| 阶段 | 演进项 | 触发条件 | 设计预留 |
| --- | --- | --- | --- |
| Phase 2 | MySQL 主从和读写分离 | 单库读 QPS 大于 5000 或备份恢复压力明显 | Repository 层封装数据访问，数据库连接由配置中心控制 |
| Phase 2 | Redis Cluster | 单实例内存大于 8 GB 或排行榜访问成为瓶颈 | Redis key 按业务域命名，避免跨 key 强事务依赖 |
| Phase 2 | RabbitMQ 集群 | 提交任务队列成为单点风险 | 消息 schema 独立在共享 API 模块，业务侧不依赖具体队列部署形态 |
| Phase 2 | Seata Server 集群 | 分布式事务演示路径需要更高可用 | Seata 客户端通过 Nacos 发现 TC 地址 |
| Phase 3 | RabbitMQ 迁移 RocketMQ 或 Kafka | 提交峰值大于 10K QPS | submit-service 通过消息发布接口隔离 MQ 实现 |
| Phase 3 | Service Mesh | 微服务数量大于 15 或治理规则复杂化 | 服务间 HTTP 接口保持标准化，便于接入 Istio |
| Phase 3 | 多沙箱 Runner | 需要 SPJ、交互题或更强隔离 | judge-worker 内部保留 Runner 和 Comparator 抽象 |
