# JudgeMesh 本地启动与演示说明

本文说明如何从零启动当前 JudgeMesh 项目，以及当前本地环境中可以稳定演示的功能范围。文档基于 2026-05-13 的仓库状态和本机验证结果编写。

## 1. 启动目标

本地启动后的目标拓扑如下：

| 类型 | 组件 | 默认地址 |
| --- | --- | --- |
| 前端 | React/Vite Web UI | `http://127.0.0.1:5173` |
| 统一入口 | Spring Cloud Gateway | `http://127.0.0.1:8080` |
| 用户服务 | `user-service` | `http://127.0.0.1:8081` |
| 题目服务 | `problem-service` | `http://127.0.0.1:8082` |
| 提交服务 | `submit-service` | `http://127.0.0.1:8083` |
| 调度器 | `judge-dispatcher` | `http://127.0.0.1:8084` |
| 判题 worker | Go judge-worker | `http://127.0.0.1:8090`，分布式演示时为 `8090/8091/8092` |
| MySQL | 业务数据库 | `127.0.0.1:3306` |
| Redis | 排行榜/缓存 | `127.0.0.1:6379` |
| RabbitMQ | 判题任务队列 | `127.0.0.1:5672` |
| RabbitMQ 控制台 | 管理面板 | `http://127.0.0.1:15672` |
| MinIO 控制台 | 对象存储面板 | `http://127.0.0.1:9001` |
| Nacos | 注册发现/配置中心 | `http://127.0.0.1:8848/nacos` |

## 2. 环境准备

在项目根目录执行：

```bash
cd /Users/phoen1xcode/Projects/dev/JudgeMesh
```

需要准备：

- Docker Desktop，并确保 Docker 已启动。
- JDK 17。
- Maven。
- Go，当前 worker 的 `go.mod` 目标为 Go 1.23。
- Node.js 与 npm。
- `curl`，用于本地接口验证。

建议先检查版本：

```bash
docker version
java -version
mvn -v
go version
node -v
npm -v
```

如果要做完全干净的从零演示，并且可以丢弃本地数据库、队列、对象存储数据，可以先清理本地中间件卷：

```bash
docker compose -f infra/local/docker-compose.yml --env-file infra/local/.env down -v
```

注意：`down -v` 会删除 MySQL、Redis、RabbitMQ、MinIO、etcd、Nacos 的本地数据卷。已有提交记录、注册用户、比赛报名等本地数据都会消失。

## 3. 启动本地中间件

首次启动先复制本地环境变量：

```bash
cp infra/local/.env.example infra/local/.env
```

启动 Docker 中间件：

```bash
docker compose -f infra/local/docker-compose.yml --env-file infra/local/.env up -d
```

检查容器状态：

```bash
docker compose -f infra/local/docker-compose.yml --env-file infra/local/.env ps
```

期望 `mysql`、`redis`、`rabbitmq`、`minio`、`etcd`、`nacos` 都处于 `Up` 或 `healthy` 状态。Nacos 首次启动会慢一些，建议等到 `jm-nacos` 变成 `healthy` 后再启动 Java 服务。

## 4. 启动后端服务

先安装服务之间共享的 API 模块：

```bash
mvn -pl services/api install -DskipTests
```

以下命令需要分别放在不同终端窗口中运行，或使用 tmux/screen 保持进程常驻。

启动用户服务：

```bash
mvn -pl services/user-service spring-boot:run
```

启动题目服务：

```bash
mvn -pl services/problem-service spring-boot:run
```

启动提交服务。本地 worker 回调地址必须指向宿主机可访问的 `127.0.0.1:8083`：

```bash
SUBMIT_CALLBACK_URL=http://127.0.0.1:8083/api/submit/internal/result \
  mvn -pl services/submit-service spring-boot:run
```

启动调度器：

```bash
mvn -pl services/judge-dispatcher spring-boot:run
```

启动网关：

```bash
mvn -pl services/gateway spring-boot:run
```

## 5. 启动 judge-worker

本机 macOS 环境通常没有 Linux `isolate` 沙箱。为了完成本地演示闭环，可以显式开启非沙箱模式：

```bash
cd services/judge-worker
LISTEN_ADDR=:8090 \
JUDGE_ALLOW_UNSANDBOXED=true \
JUDGE_WORK_DIR=/tmp \
go run ./cmd/worker
```

`JUDGE_ALLOW_UNSANDBOXED=true` 只适合本地课程演示和开发调试。生产部署应使用带 `isolate` 的 worker 容器，并关闭非沙箱模式。

worker 健康检查：

```bash
curl http://127.0.0.1:8090/health
```

期望看到：

```json
{"status":"UP","version":"0.0.1-SNAPSHOT","workerId":"worker-local"}
```

如果要给老师演示分布式调度和 worker 高可用，建议启动 3 个 worker 实例，并让 dispatcher 读取 3 个 worker URL。

先停止已有单 worker 和 dispatcher：

```bash
tmux kill-session -t jm-worker 2>/dev/null || true
tmux kill-session -t jm-worker-2 2>/dev/null || true
tmux kill-session -t jm-worker-3 2>/dev/null || true
tmux kill-session -t jm-dispatcher 2>/dev/null || true
```

启动 3 个本地 worker：

```bash
tmux new-session -d -s jm-worker \
'cd /Users/phoen1xcode/Projects/dev/JudgeMesh/services/judge-worker && LISTEN_ADDR=:8090 WORKER_ID=worker-1 JUDGE_ALLOW_UNSANDBOXED=true JUDGE_WORK_DIR=/tmp go run ./cmd/worker'

tmux new-session -d -s jm-worker-2 \
'cd /Users/phoen1xcode/Projects/dev/JudgeMesh/services/judge-worker && LISTEN_ADDR=:8091 WORKER_ID=worker-2 JUDGE_ALLOW_UNSANDBOXED=true JUDGE_WORK_DIR=/tmp go run ./cmd/worker'

tmux new-session -d -s jm-worker-3 \
'cd /Users/phoen1xcode/Projects/dev/JudgeMesh/services/judge-worker && LISTEN_ADDR=:8092 WORKER_ID=worker-3 JUDGE_ALLOW_UNSANDBOXED=true JUDGE_WORK_DIR=/tmp go run ./cmd/worker'
```

用 3 个 worker URL 启动 dispatcher：

```bash
tmux new-session -d -s jm-dispatcher \
'cd /Users/phoen1xcode/Projects/dev/JudgeMesh && JUDGE_WORKER_URLS=http://127.0.0.1:8090,http://127.0.0.1:8091,http://127.0.0.1:8092 mvn -pl services/judge-dispatcher spring-boot:run'
```

检查 worker 池和 dispatcher 配置：

```bash
for p in 8090 8091 8092; do curl http://127.0.0.1:$p/health; echo; done
curl http://127.0.0.1:8084/api/admin/dispatcher/status
```

## 9. 完整判题闭环演示

### 9.1 前端演示路径

推荐演示顺序：

1. 打开 `http://127.0.0.1:5173/login`。
2. 使用 `student@judgemesh.local` / `Student@12345` 登录。
3. 进入 `Problems`，可以看到当前种子题：
   - `A+B Problem`
   - `Echo Lines`
4. 打开 `A+B Problem`。
5. 选择 `PYTHON`，提交：

```python
a,b=map(int,input().split())
print(a+b)
```

6. 进入 `Submits` 查看提交记录，等待状态从 `PENDING`/`JUDGING` 进入 `AC`。
7. 进入首页或榜单相关页面，观察已通过提交后的统计变化。

### 9.2 API 演示脚本

也可以直接用下面的 Node 脚本演示登录、取题、提交、轮询结果的完整闭环：

```bash
node - <<'NODE'
const base = 'http://127.0.0.1:8080';

async function req(path, opts = {}) {
  const res = await fetch(base + path, {
    ...opts,
    headers: {'content-type': 'application/json', ...(opts.headers || {})},
  });
  const text = await res.text();
  if (!res.ok) throw new Error(`${opts.method || 'GET'} ${path} -> ${res.status} ${text}`);
  const json = text ? JSON.parse(text) : null;
  return json && Object.prototype.hasOwnProperty.call(json, 'data') ? json.data : json;
}

(async () => {
  const login = await req('/api/auth/login', {
    method: 'POST',
    body: JSON.stringify({
      email: 'student@judgemesh.local',
      password: 'Student@12345',
    }),
  });
  const token = login.token || login.accessToken;
  const headers = {authorization: `Bearer ${token}`};

  const problems = await req('/api/problem/list');
  const problem = problems[0];

  await new Promise((resolve) => setTimeout(resolve, 1200));

  const submit = await req('/api/submit', {
    method: 'POST',
    headers,
    body: JSON.stringify({
      problemId: problem.id,
      language: 'PYTHON',
      code: 'a,b=map(int,input().split())\nprint(a+b)\n',
    }),
  });

  let current = submit;
  for (let i = 0; i < 20; i++) {
    current = await req(`/api/submit/${submit.id}`, {headers});
    if (!['PENDING', 'JUDGING'].includes(current.status)) break;
    await new Promise((resolve) => setTimeout(resolve, 500));
  }

  console.log(JSON.stringify({
    problemId: problem.id,
    submitId: submit.id,
    status: current.status,
    score: current.score,
    worker: current.judgedByWorker,
  }, null, 2));
})();
NODE
```

期望输出类似：

```json
{
  "problemId": 1,
  "submitId": 11,
  "status": "AC",
  "score": 100,
  "worker": "worker-local"
}
```

这条路径覆盖了：

- 网关鉴权与路由。
- 用户登录与 JWT。
- 题目列表读取。
- 提交服务创建提交记录。
- RabbitMQ 投递判题任务。
- `judge-dispatcher` 消费任务并派发 worker。
- `judge-worker` 运行 Python 程序并比对输出。
- worker 回调 `submit-service`。
- 提交状态落库为 `AC`，分数为 `100`。

### 10.2 分布式调度、高负载和高可用演示

老师如果重点要求“分布式、高可用、高负载、流量暴增”，建议按下面三段演示。

#### 10.2.1 展示当前 worker 池

```bash
for p in 8090 8091 8092; do curl http://127.0.0.1:$p/health; echo; done
curl http://127.0.0.1:8084/api/admin/dispatcher/status
```

当前本地三 worker 模式下，期望看到：

- `worker-1` 运行在 `8090`。
- `worker-2` 运行在 `8091`。
- `worker-3` 运行在 `8092`。
- dispatcher 的 `configured` 字段包含 3 个 worker URL。

#### 10.2.2 模拟流量暴增并展示调度分布

使用专用演示脚本并发创建提交、轮询最终判题结果、按 worker 汇总分布：

```bash
node scripts/demo-distributed-load.mjs --total 60 --concurrency 30 --users 60
```

脚本会输出：

- `statusCounts`：各判题状态数量。
- `workerCounts`：每个 worker 实际处理了多少提交。
- `latencyMs`：端到端延迟分位数。
- dispatcher 前后状态。

当前本机已验证 30 个突发提交：

```text
statusCounts={"AC":30}
workerCounts={"worker-2":10,"worker-3":10,"worker-1":10}
```

这可以说明提交请求不是由单个 worker 处理，而是被 dispatcher 均匀派发到 worker 池。

#### 10.2.3 模拟 worker 故障和自动摘除

停止一个 worker：

```bash
tmux kill-session -t jm-worker-2
```

继续压测：

```bash
node scripts/demo-distributed-load.mjs --total 30 --concurrency 15 --users 30
```

当前本机已验证，在 `worker-2` 停止后，任务会落到剩余两个 worker：

```text
statusCounts={"AC":20}
workerCounts={"worker-1":10,"worker-3":10}
```

同时 dispatcher 状态中会出现：

```text
unavailableUntil 包含 http://127.0.0.1:8091
```

这可以说明 dispatcher 能发现故障 worker，并在黑名单时间内绕开它，剩余 worker 继续处理流量。

演示结束后恢复 `worker-2`：

```bash
tmux new-session -d -s jm-worker-2 \
'cd /Users/phoen1xcode/Projects/dev/JudgeMesh/services/judge-worker && LISTEN_ADDR=:8091 WORKER_ID=worker-2 JUDGE_ALLOW_UNSANDBOXED=true JUDGE_WORK_DIR=/tmp go run ./cmd/worker'
```

停止前端和后端进程：

- 如果使用普通终端，分别按 `Ctrl+C`。
- 如果使用 tmux，可以按实际 session 名停止，例如：

```bash
tmux kill-session -t jm-gateway
tmux kill-session -t jm-user
tmux kill-session -t jm-problem
tmux kill-session -t jm-submit
tmux kill-session -t jm-dispatcher
tmux kill-session -t jm-worker
tmux kill-session -t jm-worker-2
tmux kill-session -t jm-worker-3
tmux kill-session -t jm-frontend
```

停止 Docker 中间件但保留数据：

```bash
docker compose -f infra/local/docker-compose.yml --env-file infra/local/.env stop
```

停止并删除容器但保留数据卷：

```bash
docker compose -f infra/local/docker-compose.yml --env-file infra/local/.env down
```

停止并删除所有本地数据卷：

```bash
docker compose -f infra/local/docker-compose.yml --env-file infra/local/.env down -v
```

再次提醒：`down -v` 会清空本地数据库和中间件数据，只适合重置演示环境。
